# 仕様書・ドキュメント運用ガイド

## 前提

- 読者は「プロジェクトの文脈を一切持たない優秀なエンジニア」である。新しいチームメンバーもClaude Codeの新規セッションも同じ条件にある。
- コードから読み取れる情報（How）を自然言語で繰り返さない。ドキュメントの価値は「コードだけでは復元できない情報」にある。

---

## 1. ドキュメントの種類と責務

```
project-root/
├── CLAUDE.md                  # 環境情報・参照先の案内
├── constitution.md            # 開発上の制約・チーム規約
├── docs/
│   ├── SPECIFICATION_GUIDE.md # このファイル（ドキュメント運用ルール）
│   ├── FEATURE_SPECIFICATION.md # 機能一覧（目次）
│   ├── specs/                 # 機能仕様書
│   │   └── {feature}.md
│   ├── decisions/             # ADR (Architecture Decision Records)
│   │   └── {NNN}_{title}.md
│   └── plans/                 # 実装計画（短期）
└── src/
    └── *.{ext}                # コード内コメント / docstring
```

| 置き場所 | 役割 | 書くもの | 書かないもの |
| --- | --- | --- | --- |
| **CLAUDE.md** | オンボーディング | プロジェクト概要、技術スタック、ディレクトリ構成、ビルド/テスト/リントのコマンド、他ドキュメントへの参照 | 規約・ルール、個別機能の要件 |
| **constitution.md** | 不変のルール | チーム規約、アーキテクチャ原則、コーディング規約、テスト方針、セキュリティ要件、禁止事項 | 環境セットアップ手順、個別機能の要件 |
| **docs/specs/{feature}.md** | 機能の意図と仕様 | 目的(Why)、振る舞い(What)、制約・不変条件、システムレベルの振る舞い定義 | 実装の内部アルゴリズム（コードで表現すべき） |
| **docs/decisions/{NNN}.md** | 設計判断の記録 | コンテキスト、決定内容、理由、影響 | 実装手順（仕様書へ委譲） |
| **コード内コメント** | 設計判断の記録 | なぜこの方法を選んだか(Why this way)、非自明な制約の背景、TODO/HACKの理由、仕様書へのポインタ | コードの逐語訳、ビジネスルール（仕様書へ委譲） |

### 原則

- **各ドキュメントは1つの責務だけを持つ。** 情報の重複は矛盾の温床になる。
- **CLAUDE.md は薄く保つ。** 「何がどこにあるか」を示すハブであり、ルール自体は書かない。ルールは constitution.md へ、機能要件は仕様書へ委譲する。

---

## 2. CLAUDE.md の書き方

CLAUDE.mdは「プロジェクトに初日に来たエンジニアが最初に読む1枚」として機能させる。ルールや要件への**ポインタ**であり、ルールそのものではない。

**100行を大幅に超えたら分割を検討する。** サブディレクトリごとの CLAUDE.md に技術的な詳細を分けてよい。

---

## 3. constitution.md の書き方

constitution.md は「プロジェクト全体を貫く、個別機能に依存しないルール」を集約する。

### 記述ルール

- **各ルールに「なぜそのルールがあるのか」を併記する。** 理由のないルールはAIにも人間にも軽視される。
- **NON-NEGOTIABLE（交渉不可）な項目を明示する。** 状況に応じて柔軟に判断してよい項目と区別する。
- **定期的に棚卸しする。** 形骸化したルールは削除する。ルールの数が増えるほど遵守率は下がる。

---

## 4. 仕様書 (docs/specs/{feature}.md) の書き方

### セクション構成

仕様書は「**契約**（上位）」と「**実装ガイド**（下位）」の2層で構成する。

```markdown
# {機能名}

<!-- 上位: 契約 — この機能が「なぜ」「何を」「何を変えてはいけないか」 -->

## 目的（Why）
この機能がなぜ存在するのか。解決するユーザーの課題。

## 振る舞い（What）
ソフトウェアが外部から見てどう動くかを記述する。
Given/When/Then 形式、または入出力の具体例で書く。

## 制約・不変条件（Boundaries）
変えてはいけないこと。外部との契約。
「なぜ変えてはいけないのか」の理由も必ず添える。

## 未決事項（Open Questions）（任意）
まだ判断していないこと。仮決定とその根拠。

<!-- 下位: 実装ガイド — エージェントがそのまま実装できるレベルの詳細 -->

## バックエンドコマンド
Tauriコマンドの一覧

## データモデル
Rust構造体、TypeScript型定義

## フロントエンド
UIコンポーネントとユーザー操作

## 永続化
保存先とデータ形式

## エラーハンドリング
エラーケースと対応

## 運用ケース
実際の使用シナリオ

## マイグレーション（必要な場合のみ）
旧バージョンからの移行手順
```

### 記述ルール

- **「何をするか」ではなく「ユーザーから見てどう振る舞うか」で書く。** 内部処理のステップではなく、観測可能な結果を記述する。
- **具体例を必ず含める。** 抽象的な説明だけでは、AIも人間も異なる解釈をする。正常系1つ＋異常系1つが最低ライン。
- **暗黙の要件を明示する。** 「当然こうだろう」と思うことほど書く。パフォーマンス要件、順序保証、冪等性など。
- **変更可能な部分と不可変な部分を区別する。** 「要件として固定」なのか「現時点の設計判断で変更可能」なのかが分かるようにする。

### 書き方のポイント

#### 明確な表現

```markdown
# 良い例
認証情報はWindows Credential Managerに保存する。

# 悪い例
認証情報は安全な場所に保存することが望ましい。
```

#### 表形式の活用

```markdown
# 良い例
| 状態 | 表示 |
|-----|------|
| 未認証 | グレー |
| 認証済み | 緑 |

# 悪い例
未認証の場合はグレー、認証済みの場合は緑で表示する。
```

#### コード例は具体的に

```markdown
# 良い例
**保存形式（JSON）:**
{
  "sid": "xxx",
  "sapisid": "xxx"
}

# 悪い例
JSON形式で保存する。
```

## 5. ADR (Architecture Decision Records) の書き方

### ルール

1. **1つの判断につき1ファイル**
2. **番号付きで管理** — `001_`, `002_`, ...
3. **変更しない** — 判断を覆す場合は新しいADRを作成
4. **コンテキストと理由を明記** — テンプレートは `docs/decisions/_TEMPLATE.md` を参照

### 仕様変更時のフロー

```
1. 新しい設計判断がある場合
   → decisions/ にADRを作成

2. specs/ の該当仕様書を更新
   → 旧仕様を削除し、新仕様に置き換え
   → マイグレーションが必要ならセクション追加

3. gitでコミット
   → 変更履歴はgitで追跡
```

## 6. コード内コメント / docstring の書き方

### 書くべきもの

- **Why this way: なぜこの方法を選んだか。** 却下した代替案とその理由があるとなお良い。
- **非自明な制約:** 外部仕様やバグ回避に起因するコード上の制約。将来の開発者やAgentが「これ消しても良さそう」と思いそうな箇所。
- **TODO / HACK の背景:** 一時的な回避策には、なぜそうしたか・いつ解消すべきかを書く。
- **仕様書へのポインタ:** ビジネスルールに関わるロジックには、該当する仕様書への参照を書く（例: `// See: docs/specs/04_tts.md#振る舞い`）。

### 書かないもの

- コードを読めば分かること（`i += 1  // iを1増やす` のような逐語訳）。
- 仕様書に書くべきビジネスルール（二重管理になる。ポインタだけを書く）。

---

## 7. 運用上の注意

- **仕様書は「生きたドキュメント」にする。** 実装と乖離した仕様書は害悪。機能を変更したら仕様書も同時に更新する。
- **完璧を目指さない。** 最初から全てを書こうとしない。仕様書は「目的」「振る舞いの具体例」「変えてはいけないこと」の3点さえあれば最低限機能する。
- **定期的に棚卸しする。** 仕様書が実装と一致しているか、constitution.md のルールが形骸化していないかを確認する機会を設ける。
- **ドキュメント更新を実装とセットにする。** PRやコミットの単位で、関連ドキュメントの更新を含める習慣をつける。
