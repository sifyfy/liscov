# 仕様書ガイド

## ディレクトリ構造

```
docs/
├── FEATURE_SPECIFICATION.md     # 機能一覧（目次）
├── SPECIFICATION_GUIDE.md       # このファイル（仕様書の書き方）
├── specs/                       # 機能仕様書（最新仕様のみ）
│   ├── 01_auth.md
│   ├── 02_chat.md
│   └── ...
└── decisions/                   # ADR (Architecture Decision Records)
    ├── _TEMPLATE.md             # ADRテンプレート
    ├── 001_secure_storage.md
    └── ...
```

## 仕様書の原則

### specs/ ディレクトリ

**目的:** コーディングエージェントが実装時に参照する

**ルール:**
1. **常に最新仕様のみを記載** - 過去の仕様は記載しない
2. **一貫性を保つ** - 矛盾する記述を含めない
3. **実装可能な粒度で記載** - エージェントがそのまま実装できるレベル
4. **マイグレーションは専用セクションで** - 旧仕様からの移行が必要な場合のみ記載

**セクション構成（推奨）:**
```markdown
# 機能名

## 概要
機能の目的と概要

## バックエンドコマンド
Tauriコマンドの一覧

## フロントエンド
UIコンポーネントとユーザー操作

## 永続化
保存先とデータ形式

## エラーハンドリング
エラーケースと対応

## 運用ケース
実際の使用シナリオ

## マイグレーション（必要な場合のみ）
旧バージョンからの移行手順
```

### decisions/ ディレクトリ

**目的:** 設計判断の経緯を記録（人間用）

**ルール:**
1. **1つの判断につき1ファイル**
2. **番号付きで管理** - `001_`, `002_`, ...
3. **変更しない** - 判断を覆す場合は新しいADRを作成
4. **コンテキストと理由を明記**

## 仕様変更時のフロー

```
1. 新しい設計判断がある場合
   → decisions/ にADRを作成

2. specs/ の該当仕様書を更新
   → 旧仕様を削除し、新仕様に置き換え
   → マイグレーションが必要ならセクション追加

3. gitでコミット
   → 変更履歴はgitで追跡
```

## 書き方のポイント

### 明確な表現
```markdown
# 良い例
認証情報はWindows Credential Managerに保存する。

# 悪い例
認証情報は安全な場所に保存することが望ましい。
```

### 表形式の活用
```markdown
# 良い例
| 状態 | 表示 |
|-----|------|
| 未認証 | グレー |
| 認証済み | 緑 |

# 悪い例
未認証の場合はグレー、認証済みの場合は緑で表示する。
```

### コード例は具体的に
```markdown
# 良い例
**保存形式（JSON）:**
```json
{
  "sid": "xxx",
  "sapisid": "xxx"
}
```

# 悪い例
JSON形式で保存する。
```
