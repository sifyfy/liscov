# ADR-003: 仕様駆動テスト原則

## ステータス

承認

## コンテキスト

TTS機能の `strip_handle_suffix` が仕様（末尾の `-xxx` サフィックス除去）と異なる実装（`" @"` パターン検索）になっていたが、ユニットテストが全てパスしていたため長期間検出されなかった。

### 発見された構造的問題

1. **仕様-実装の乖離**: 仕様の定義（`-xxx` 除去）と実装のロジック（`" @"` 検索）が完全に異なっていた
2. **コード重複**: 同一ロジックが `format_text()` と `start_processing()` に複製されており、片方を修正しても漏れるリスクがあった
3. **テストが実装を検証**: テストが仕様の例ではなく、実装の（誤った）動作を検証していた。テストは通るが仕様は満たされない状態
4. **抽出ヘルパー未使用**: テスト容易性のために抽出した純粋関数が本番コードから呼ばれず、テストの検証対象と本番コードパスが乖離していた

## 決定

以下の3原則を開発ルールとして採用する。

### 原則1: テストケースは仕様の具体例から導出する

テストの期待値は仕様書（`docs/specs/*.md`）の具体例テーブルから直接転記する。実装を見てテストを書かない。

```rust
// GOOD: 仕様テーブルの例をそのまま転記
#[test]
fn spec_example_all_options_on() {
    // spec (04_tts.md line 171): @田中-abc → 田中さん
    assert_eq!(process_author_name("@田中-abc", true, true, true), "田中さん");
}

// BAD: 実装を動かして出力をコピペ
#[test]
fn test_strip() {
    assert_eq!(process_author_name("@田中-abc", true, true, true), "田中-abcさん");
}
```

### 原則2: 抽出した関数は必ず本番コードから呼び出す

テスト容易性のために抽出した純粋関数は、元の場所から必ず呼び出す。テスト用コピーが本番パスと乖離することを防ぐ。

```rust
// GOOD: 本番コードがヘルパーを呼ぶ
pub async fn format_text(&self, item: &TtsQueueItem) -> String {
    let config = self.config.read().await;
    build_tts_text(item.author_name.as_deref(), /* ... */)
}

// BAD: ヘルパーが存在するが本番コードは独自実装
pub async fn format_text(&self, item: &TtsQueueItem) -> String {
    let mut parts = Vec::new();
    // ... ヘルパーと同じロジックを手書き
}
```

### 原則3: ロジック重複の禁止

同一ビジネスロジックの複製を禁止。関数抽出 + 呼び出しで一元化する。

## 理由

### 検討した選択肢

| 選択肢 | メリット | デメリット |
|-------|---------|-----------|
| A: 仕様駆動テスト原則（採用） | 仕様-実装の乖離を検出可能 | テスト作成時に仕様参照が必要 |
| B: プロパティベーステスト | 広範囲の入力をカバー | 仕様の「意味」は検証できない |
| C: E2Eテストのみで検証 | 本番に近い検証 | 実行コスト高、デバッグ困難 |

### 採用理由

- 仕様書の具体例テーブルはそのままテストケースになるため追加コストが低い
- 仕様-実装の意味的乖離（パターン違い等）を最も直接的に検出できる
- ユニットテストレベルで検出でき、フィードバックが高速

## 影響

- 今後のテスト作成時、仕様の具体例を先に確認する習慣が必要
- 既存テストの一部を仕様例ベースに修正（本ADR適用時に実施済み）
- `format_text` / `start_processing` の重複を解消し `build_tts_text` に一元化（実施済み）
