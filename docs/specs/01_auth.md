# 認証機能

## 目的（Why）

配信者がYouTubeのメンバー限定配信のチャットにアクセスするには、YouTubeアカウント認証が必要である。本機能はWebViewベースのログインフローでYouTube認証情報を取得・管理し、メンバー限定コンテンツへのアクセスを可能にする。

## 振る舞い（What）

### 認証フロー

| 操作 | 結果 |
|-----|------|
| ユーザーが「YouTubeにログイン」をクリック | WebViewウィンドウが開き、YouTubeログイン画面が表示される |
| WebViewでGoogleアカウントにログイン完了 | SAPISID Cookieを検出し、認証情報をセキュアストレージに自動保存。ウィンドウが自動で閉じる |
| 5分以内にログイン完了しない | タイムアウトエラーを表示 |
| ユーザーがウィンドウを閉じる | 認証キャンセルとして処理 |

### ログアウトフロー

| 操作 | 結果 |
|-----|------|
| ユーザーが「ログアウト」をクリック | セキュアストレージから認証情報を削除。WebViewのCookieもクリア |

### セッション有効性検証

| タイミング | 結果 |
|-----------|------|
| アプリ起動時（認証情報あり） | バックグラウンドでYouTube APIにテストリクエストを送信し、有効性を確認 |
| 有効 | インジケーター: 緑（「ログイン中: 有効」） |
| 無効（401/403） | インジケーター: 赤（「セッション切れ - 再ログインが必要」） |
| ネットワークエラー | インジケーター: オレンジ（「検証失敗（ネットワークエラー）」） |

### ストレージ障害時

| 状況 | 結果 |
|------|------|
| セキュアストレージにアクセスできない | ダイアログで通知。ユーザーが「ファイル保存を使用」を選択した場合のみfallbackに切り替え |
| ユーザーの明示的な同意なし | 自動フォールバックはしない |

## 制約・不変条件（Boundaries）

| 制約 | 理由 |
|------|------|
| 認証情報はセキュアストレージ（Windows Credential Manager）を優先し、ユーザーの明示的な同意なしにファイル保存にフォールバックしない | セキュリティ上、暗号化されていないファイルへの自動降格は許容できない（ADR-001） |
| SAPISIDHASH生成には必ず5つのCookie（SID, HSID, SSID, APISID, SAPISID）が必要 | YouTube InnerTube APIの認証要件。1つでも欠けるとAPI呼び出しが失敗する |
| SAPISIDHASH = `SHA1(timestamp + " " + SAPISID + " " + origin)` の形式 | YouTube APIとの契約。変更するとメンバー限定配信にアクセスできなくなる |
| 認証ウィンドウのタイムアウトは300秒（5分） | 2FA認証を含むログインフローに十分な時間を確保しつつ、放置ウィンドウを防ぐ |
| E2Eテストは環境変数 `LISCOV_KEYRING_SERVICE` でサービス名を分離する | 本番の認証情報をテストが破壊することを防ぐ |

## バックエンドコマンド

| コマンド | 入力 | 出力 | 説明 |
|---------|------|------|------|
| `auth_get_status` | なし | `AuthStatus { is_authenticated, has_saved_credentials, storage_type, storage_error? }` | 認証状態を取得 |
| `auth_open_window` | `app: AppHandle` | `()` | WebViewログインウィンドウを開く |
| `auth_delete_credentials` | なし | `()` | 保存済み認証情報を削除 |
| `auth_clear_webview_cookies` | `app: AppHandle` | `()` | WebViewのCookieをクリア（YouTubeからログアウト） |
| `auth_load_credentials` | なし | `bool` | 認証情報をロード |
| `auth_validate_credentials` | なし | `bool` | 認証情報の形式を検証（ローカルチェックのみ） |
| `auth_save_raw_cookies` | `raw_cookies: String` | `()` | Raw cookie文字列を保存 |
| `auth_save_credentials` | `sid, hsid, ssid, apisid, sapisid: String` | `()` | 個別認証情報を保存 |
| `auth_check_session_validity` | なし | `SessionValidity { is_valid, checked_at, error? }` | セッションの有効性を検証（YouTube APIへのテストリクエスト） |
| `auth_use_fallback_storage` | なし | `bool` | ストレージモードをfallbackに切り替え（config.tomlに保存） |

### 廃止コマンド

| コマンド | 理由 |
|---------|------|
| `auth_get_credentials_path` | セキュアストレージ使用により不要。storage_typeで確認可能 |

> **Note**: `auth_clear_webview`は`auth_clear_webview_cookies`として復活しました（Tauri v2の`clear_all_browsing_data()` APIを使用）。

|----|-----------|------|

**保存形式（JSON）:**
```json
{

  "hsid": "xxx",
  "ssid": "xxx",
  "apisid": "xxx",
  "sapisid": "xxx"

```

## 認証ウィンドウ仕様

| 項目 | 値 |
|-----|-----|
| 初期URL | `https://www.youtube.com/` |
| ウィンドウサイズ | 900 x 700 px |
| タイムアウト | 300秒（5分） |
| Cookieポーリング間隔 | 1000ms（1秒） |
| ログイン完了判定 | SAPISID Cookie検出 |

## 必須Cookie

YouTube API認証には以下の5つのCookieが必須：

| Cookie名 | 用途 |
|---------|------|
| SID | セッションID |
| HSID | HTTPセキュアセッションID |
| SSID | セキュアセッションID |
| APISID | API用セッションID |
| SAPISID | セキュアAPI用セッションID（SAPISIDHASH生成に使用） |

## SAPISIDHASH生成

メンバー限定配信へのアクセスに必要な認証ヘッダー。

**生成ロジック:**
```
入力: SAPISID, タイムスタンプ, オリジン("https://www.youtube.com")
処理: SHA1(タイムスタンプ + " " + SAPISID + " " + オリジン)
出力: "{タイムスタンプ}_{SHA1ハッシュ(40文字hex)}"
```

**認証ヘッダー:**
```
Authorization: SAPISIDHASH {生成した値}
Cookie: SID=...; HSID=...; SSID=...; APISID=...; SAPISID=...
X-Origin: https://www.youtube.com
Origin: https://www.youtube.com
```

## 認証有効性テスト

### 概要

保存されている認証情報が実際にYouTube APIで使用可能かをバックグラウンドで検証する。

### 検証方法

YouTube InnerTube APIに軽量なリクエストを送信し、認証が有効かを確認する。

**検証エンドポイント:**
- `https://www.youtube.com/youtubei/v1/account/account_menu`
- 認証ヘッダー付きでリクエストし、正常レスポンスを確認

### 検証タイミング

| タイミング | 条件 | 動作 |
|-----------|------|------|
| アプリ起動時 | 認証情報が存在する場合 | バックグラウンドで検証開始 |
| 設定画面を開いた時 | 手動で「検証」ボタンをクリックした場合 | 即時検証 |
| 配信接続時 | 認証情報が存在する場合 | 接続処理の一部として検証（必須） |

### 検証結果

| 結果 | HTTPステータス | 動作 |
|-----|---------------|------|
| 有効 | 200 | インジケーターを「認証済み（有効）」に更新 |
| 無効（認証エラー） | 401 | インジケーターを「認証済み（無効/期限切れ）」に更新 |
| 無効（権限エラー） | 403 | インジケーターを「認証済み（無効/期限切れ）」に更新 |
| ネットワークエラー | - | インジケーターを「検証エラー」に更新 |

### Tauri Event

| イベント名 | ペイロード | 説明 |
|-----------|-----------|------|
| `auth-session-validity-changed` | `{ is_valid: bool, error?: string }` | セッション有効性が変更された時に発行 |

## エラーハンドリング

### 認証ウィンドウエラー

| エラー種別 | 発生条件 | ユーザーへの表示 |
|-----------|---------|-----------------|
| WindowCreation | ウィンドウ作成失敗 | 「認証ウィンドウの作成に失敗しました」 |
| Timeout | 5分以内にログイン完了しなかった | 「認証がタイムアウトしました。再度お試しください」 |
| Cancelled | ユーザーがウィンドウを閉じた | 「認証がキャンセルされました」 |
| CookieExtraction | Cookie抽出・保存失敗 | 「認証情報の取得に失敗しました」 |

### HTTP認証エラー（チャット取得時）

| ステータス | 意味 | 動作 |
|-----------|-----|------|
| 401 Unauthorized | Cookie期限切れまたは無効 | インジケーターを「無効/期限切れ」に更新、処理継続 |
| 403 Forbidden | メンバー限定コンテンツへのアクセス権限なし | インジケーターを「無効/期限切れ」に更新、処理継続 |
| 429 Too Many Requests | レート制限 | メッセージ取得失敗としてログ、処理継続 |

## 運用ケース

### アプリ起動時の動作

| ステップ | 処理 |
|---------|------|
| 1 | セキュアストレージから認証情報をロード |
| 2 | 認証情報が存在する場合、インジケーターを「検証中」に設定 |
| 3 | バックグラウンドで`auth_check_session_validity`を実行 |
| 4 | 検証結果に応じてインジケーターを更新 |

### ネットワークエラー時の動作

| 状況 | 動作 |
|-----|------|
| 認証ウィンドウでネットワーク断 | WebViewがエラーページを表示。タイムアウトまで待機後、Timeout エラー |
| チャット取得中にネットワーク断 | メッセージ取得失敗としてログ。ポーリング継続、復旧時に自動再取得 |
| 有効性検証中にネットワーク断 | インジケーターを「検証エラー」に更新 |

### Googleアカウント2段階認証（2FA）

WebViewはフルブラウザとして動作するため、以下の2FA方式に対応：
- SMS認証コード
- Google認証システムアプリ
- セキュリティキー（WebAuthn）
- Googleプロンプト（別端末での承認）

### 配信接続中の認証情報変更

| 操作 | 動作 |
|-----|------|
| 接続中にログアウト | 既存のInnerTubeクライアントは旧認証情報を保持。メンバー限定チャットの取得は継続可能 |
| 接続中に再ログイン | 既存接続には影響なし。次回の`connect_to_stream`から新しい認証情報を使用 |
| Disconnect後に再接続 | 最新の認証情報を読み込んで接続 |

## ストレージモード

認証情報の保存先を決定するモード。

| モード | 保存先 | 条件 |
|-------|-------|------|
| secure | Windows Credential Manager | デフォルト。セキュアストレージが利用可能 |
| fallback | credentials.toml | ユーザーがファイル保存を選択した場合 |

### モードの永続化

ストレージモードは `config.toml` に保存する。詳細は[設定機能仕様](09_config.md)を参照。

## 認証情報ロード処理

### 処理フロー

```
1. config.tomlからストレージモードを確認
   ├─ fallbackモード → credentials.tomlから読み込み（終了）
   └─ secureモード（デフォルト）
        ↓
2. セキュアストレージから認証情報を読み込み試行
   ├─ 成功 → 認証情報を使用、有効性検証へ（終了）
   └─ 失敗
        ↓
3. 失敗の原因を判定
   ├─ 認証情報なし → 4へ（マイグレーション確認）
   └─ ストレージ障害 → 5へ（ユーザー確認）
        ↓
4. credentials.tomlが存在するか確認（マイグレーション）
   ├─ 存在しない → 未認証状態（終了）
   └─ 存在する → セキュアストレージへ移行、成功したらtoml削除（終了）
        ↓
5. ストレージ障害通知ダイアログを表示
   ├─ 「ファイル保存を使用」→ fallbackモードに切り替え、credentials.tomlから読み込み
   ├─ 「設定を開く」→ 設定画面を開く、未認証状態
   └─ 「無視」→ 未認証状態（終了）
```

### ポイント

| 項目 | 説明 |
|-----|------|
| モード優先 | 保存されたストレージモードに従って読み込み先を決定 |
| ユーザー確認必須 | ストレージ障害時は自動フォールバックせず、ユーザーの明示的な同意を得る |
| マイグレーション | secureモードで認証情報なし、かつcredentials.tomlが存在する場合は自動移行 |

## 認証情報保存処理

### 処理フロー

```
1. config.tomlからストレージモードを確認
   ├─ fallbackモード → credentials.tomlに保存（終了）
   └─ secureモード
        ↓
2. セキュアストレージへの保存を試行
   ├─ 成功 → 保存完了（終了）
   └─ 失敗
        ↓
3. ストレージ障害通知ダイアログを表示
   ├─ 「ファイル保存を使用」→ fallbackモードに切り替え、credentials.tomlに保存
   └─ 「キャンセル」→ 保存失敗、エラーをユーザーに通知
```

### ポイント

| 項目 | 説明 |
|-----|------|
| モードに従う | 保存先はストレージモードによって決定 |
| ユーザー確認必須 | secureモードで保存失敗時も、ファイル保存への切り替えはユーザー確認が必要 |

## E2Eテスト

### テスト分離

E2Eテストは本番データと分離された専用の名前空間を使用する。以下の環境変数でテスト用の名前空間を指定可能。

| 環境変数 | デフォルト値 | テスト時の値 | 影響範囲 |
|---------|-------------|-------------|---------|
| `LISCOV_APP_NAME` | `liscov` | `liscov-test` | 設定ファイルパス、DBパス |
| `LISCOV_KEYRING_SERVICE` | `liscov` | `liscov-test` | Windows資格情報マネージャーのサービス名 |
| `LISCOV_AUTH_URL` | YouTube URL | mock server URL | 認証ウィンドウの初期URL |
| `LISCOV_SESSION_CHECK_URL` | YouTube API | mock server URL | セッション検証エンドポイント |

### テストデータのパス

テスト実行時（`LISCOV_APP_NAME=liscov-test`）のデータ保存先：

| 種別 | パス |
|-----|------|
| 設定ファイル | `%APPDATA%/liscov-test/config.toml` |
| 認証情報（fallback） | `%APPDATA%/liscov-test/credentials.toml` |
| データベース | `%APPDATA%/liscov-test/liscov.db` |
| 認証情報（secure） | Windows資格情報マネージャー（サービス名: `liscov-test`） |

### テストスイート

E2Eテストは `e2e/auth-flow.spec.ts` に実装。

| テストケース | 検証内容 |
|-------------|---------|
| Initial State | 初回起動時に未認証状態が表示されること |
| Login Flow | WebViewログイン後にログアウトボタンが表示されること |
| AuthIndicator | ログイン後にインジケーターが「認証済み」に変わること |
| Persistence | アプリ再起動後も認証状態が維持されること |
| Logout Flow | ログアウト後にログインボタンが再表示されること |
| Cookie Clearing | ログアウト後の再ログインでログインページを訪問すること（Cookie削除確認） |

### モックサーバー

認証テスト用のモックサーバー（`src-tauri/src/bin/mock_server.rs`）を提供。

**認証関連エンドポイント:**

| エンドポイント | メソッド | 説明 |
|--------------|---------|------|
| `/` | GET | ログインページ（`auto_login=true`で自動ログイン） |
| `/do_login` | POST | ログイン処理（Cookie設定） |
| `/logged_in` | GET | ログイン完了ページ |
| `/youtubei/v1/account/account_menu` | POST | セッション検証エンドポイント |
| `/set_auth_state` | POST | 認証状態を制御（テスト用） |
| `/auth_status` | GET | 現在の認証状態を取得 |
| `/status` | GET | サーバー状態（`login_page_visits`含む） |
| `/reset` | POST | サーバー状態をリセット |

**Cookie削除の検証方法:**

`login_page_visits` カウンターを使用してCookie削除を検証：
1. `/reset` でカウンターをリセット
2. ログアウト実行
3. 再ログイン実行
4. `/status` で `login_page_visits > 0` を確認

Cookieが正しく削除されていれば、再ログイン時にログインページを訪問する（カウンター増加）。
Cookieが残っていれば、既存のSAPISIDを検出してログインページをスキップする（カウンター=0）。
